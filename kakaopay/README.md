## < 문제 해결을 위한 고려 사항 >
### 1."uid" 생성 관련
    - DB 시퀀스를 또는 고유 값을 부여 하는 로직이 필요
    - 타임스탬프(yyyymmddHHMMSS) + 랜덤숫자(6자리) 를 결합하여 중복을 방지

### 2. 파라미터(Parameter) 값 검증 관련
    - 프로그램 실행을 위한 데이터에 대한 유효성 검증( eg type, 값 유무 등 ) 및 필수 파라미터에 대한 값검증 필요
    - 카드 및 결제정보 관련 값검증 수행 사항 예시
        ● 카드번호
        ● cvc
        ● 결제금액 등
  
### 3. 부가가치세 및 부분취소관련
    - 로직 구현을 위해 총결제액 및 부가가치세 잔액에 대한 실제 운영시 발생 가능한 여러 사례에 대한 가정 필요
    - 특히, 하나의 결제이력에 대한 다수의 부분취소 이력이 발생한 경우 중 예외처리 반영여부에 대한 처리사항 고려
        ● "총결제금액/11≠부가가치세 총취소요청액" 인 경우 
        ●  최종 취소 후, 총결제액 및 부가가치세 잔액 중 하나의 잔액은 모두 소진 되었으나 
           다른 하나의 잔액의 소액이 발생
    - 일반적으로, 부가가치세란 상품에 대한 세금이나 실제 서비스 운영시 부분취소로직 적용 이후 상품에 대한 금액이 0 이여도 
      부가가치세 일부만으로도 결제나 청구 요청이 발생하는지 고민함
       
       
## < 문제 수행 사항 >
### 1. "응답값"
    - 프로그램 수행 후 결과요청 값
       ● 요청 성공 : 0000
       ● 요청 실패 : 그외에 값이 응답된 경우 
 
### 2. 개발환경
    - 문제 수행을 위해 사용된 프로그램 개발 환경
       ● 프레임워크 : String boot
       ● DB : mysql
       ● 테이블 스크립트 : table_script.txt

### 3. Test Case 
    - Case1) 결제요청 
      : http://127.0.0.1/card/setPaying?cardNo=[카드번호]&payment=[결재금액]&mmyy=[유효기간]&cvc=[cvc]&installment=[할부개월]
    - Case2) 결재 부분, 전체취소
      : http://127.0.0.1/card/setCancel?uid=[결제요청UID]&payment=[취소금액]&vat=[취소부가가치세]
    - Case3) 결제 요청 정보
      : http://127.0.0.1/card/getPayment?uid=[응답UID]
